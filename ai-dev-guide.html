2. Why We Chose Cursor — And Why One IDE Matters

3. Where Claude Fits — Model vs. IDE

Claude is the AI model (by Anthropic). Cursor is the IDE that runs Claude inside it. These aren't competing choices — they're complementary layers. Claude Opus 4.6 is the most advanced coding model available today, excelling at deep reasoning, cross-file architecture, complex debugging, and production-quality code generation. We use it heavily.

The question is how you access Claude: through an IDE (Cursor) or through a standalone terminal agent (Claude Code CLI).

Cost$20–40/user/mo (flat)Usage-based ($20–100+/mo)

Our Approach: Cursor as the IDE, Claude as the Brain

Cursor is the standard IDE for daily development — visual workflow, team consistency, lower learning curve. Claude Opus 4.6 runs inside Cursor as the primary model for complex tasks. Claude Code CLI is available for senior developers on specific use cases: large-scale multi-repo refactoring, CI/CD automation, and infrastructure work where terminal-native workflows are more natural.

For clients: If your team uses Claude-powered tools separately, we recommend allocating dedicated API quotas to avoid rate-limiting during heavy development sprints. Happy to advise on configuration.

4. The Real Problem: Proficiency, Not Tools

The most important finding: most developers use AI tools at a basic level, regardless of which tool. The gap between "using AI" and "using AI effectively" is where the real productivity unlock lives.

FeatureAdoptionAssessment
@ File References80%Good baseline
Multi-file Editing77%Healthy
Terminal Commands via AI73%Solid
Plan / Think Mode50%Half the team skips structured planning
Context Files (.cursorrules)30%Critical gap — #1 priority

Context files at 30% is the single biggest issue. Without one, every AI interaction starts from scratch — the AI doesn't know your stack, conventions, or what files to avoid. With it, the AI behaves like a developer who has read your project docs.

Plan Mode at 50% means half the team jumps to code generation without making the AI think first. For tasks touching 3+ files, Plan Mode forces the AI to outline changes, intent, and risks before writing code.

5. Three Practices That Actually Move the Needle

Any engineering team can adopt these regardless of which AI tool they use.

Context Files

What: A config file (e.g. .cursorrules) committed to your repo that tells the AI your stack, architecture, naming conventions, and security boundaries.

Why: Without it, AI guesses — React patterns in an Angular project, snake_case where you use camelCase. With it, every suggestion aligns with your conventions.

How: We've open-sourced a prompt that scans your repo and generates one automatically.

Plan Mode

What: AI produces a plan before writing code: which files change, intended edits, risks. You review before execution.

Why: Without planning, AI modifies wrong files or introduces side effects. With it, the AI thinks before it acts.

Rule: If a task touches 3+ files, always Plan Mode first.

Model Selection

What: Deliberately choosing which AI model based on task complexity, not relying on "Auto."

Why: Auto picks fast but less capable models. Complex debugging needs Claude Opus. Quick edits need Sonnet/Haiku.

Rule: Opus for complex work. Faster models for routine. Never Auto for important tasks.

6. Security Baseline

AI tools can read and expose sensitive files if not configured. Our context files enforce explicit exclusion rules per project.

What We Protect

  • Environment files (.env, .env.*)
  • API keys, secrets, credentials
  • Private keys (*.pem, *.key)
  • Auth modules and auth configs
  • Database migration files
  • CI/CD pipeline configs
  • Lock files (package-lock.json, yarn.lock)
  • Generated/vendor code (dist/, node_modules/)

How It Works

Context files include a "protected areas" section that tells the AI: never read, modify, or reference these files. Separate from .gitignore — .gitignore affects Git, not what AI accesses in your working directory.

Also enforces: "never hardcode secrets", "use env vars for all config", "ask before modifying auth/ directory."

Key: Configured per-project, committed to repo — applies to every developer automatically.
